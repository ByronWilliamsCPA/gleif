## Patterns Observed

- **OpenSSF required files absent**: All five of LICENSE, SECURITY.md, CONTRIBUTING.md, CHANGELOG.md, and README.md (empty) were missing or empty. This is the most common first-run failure pattern for repos that were bootstrapped from a minimal template without a compliance pass. Appeared in 5 of 5 required-file checks (5/5 checks failing).

- **Mutable tags in pre-commit config**: All 5 rev values in `.pre-commit-config.yaml` were mutable tags rather than SHA pins. This is a supply-chain risk that is easy to miss because pre-commit does not warn on mutable revs at install time.

- **Bandit `|| true` in CI**: The security scan step in `ci.yml` appended `|| true`, causing all Bandit findings to be silently swallowed. CI showed green while the scanner found real issues. This pattern provides false assurance and is worth a dedicated manifest check.

- **Unpinned action SHAs in CI**: Two actions (`qltysh/qlty-action/coverage@v2` and `qltysh/qlty-action@v2`) used mutable version tags. This mirrors the pre-commit mutable-tag pattern at the GitHub Actions layer.

- **BasedPyright in standard mode**: The project had BasedPyright configured in `standard` mode rather than `strict`. The toolchain-auditor found this important enough to remediate; strict mode surfaces a class of type errors that standard mode hides.

- **Missing harden-runner in CI jobs**: Three jobs (`detect-changes`, `security-gate`, `ci-gate`) lacked `step-security/harden-runner`, leaving those jobs without network egress monitoring.

- **CLAUDE.md missing required sections**: Model Selection, RAD, and Compact Instructions sections were all absent from the project CLAUDE.md. These are required by the global standard but are easy to omit when CLAUDE.md is authored before the global standard is finalised.

- **Em-dashes in documentation**: 19 em-dash characters were found across `CLAUDE.md` and `docs/index.md`. This violates the hard rule in the global writing standard and was not caught earlier because pre-commit did not yet enforce it. The no-em-dash pre-commit hook was added as part of remediation.

- **Branch protection absent**: Branch protection was not configured on `main`. This is a common pattern for solo-maintainer repos where the developer is the only committer and treats the protection as an unnecessary friction. The OSSF Scorecard penalises this regardless of team size.

- **Scorecard workflow absent**: No `.github/workflows/scorecard.yml` existed, meaning the repo was invisible to OpenSSF Scorecard automated scoring.

## Proposed Manifest Additions

### FOUND-005b: Empty README at project root

Projects using MkDocs or similar doc generators often treat `docs/index.md` as the primary entry point and leave `README.md` empty or with a one-line placeholder. GitHub visitors see a blank repository description. Proposed check:

```yaml
- id: FOUND-005b
  domain: foundations
  severity: critical
  description: "README.md is present but empty or contains only a placeholder (< 50 non-whitespace characters)"
  verify: "file_min_content: README.md, 50"
  override_eligible: false
  notes: "Distinct from FOUND-005 (absent). Catches repos that pass file_exists but provide no content."
```

### CI-007a: Security scanner silenced with `|| true`

Appending `|| true` to a SAST or dependency scanner in CI causes the job to pass even when the scanner finds real issues. The scanner is present but provides false assurance. Proposed check:

```yaml
- id: CI-007a
  domain: ci
  severity: important
  description: "Security scanner step in CI uses '|| true' or equivalent, silencing all findings"
  verify: "grep_absent: .github/workflows/*.yml, '|| true'"
  override_eligible: false
  notes: >
    Covers Bandit, osv-scanner, pip-audit, and similar tools. Acceptable alternative:
    scope the scanner to --level HIGH so only high-severity findings block CI, but do
    not suppress the exit code entirely.
```

### CI-HARDEN-001: harden-runner absent from non-build jobs

Jobs that do not build artifacts (gates, detection jobs, notification jobs) are commonly missed when harden-runner is added because authors focus on the jobs that actually run code. Proposed check:

```yaml
- id: CI-HARDEN-001
  domain: ci
  severity: important
  description: "One or more CI jobs are missing step-security/harden-runner as the first step"
  verify: "all_jobs_have_step: .github/workflows/*.yml, step-security/harden-runner"
  override_eligible: true
  notes: >
    Jobs that use 'runs-on: ubuntu-latest' or any GitHub-hosted runner should include
    harden-runner. Override is eligible for jobs that run in a fully isolated
    self-hosted environment with network egress already controlled at the infra level.
```

### ARCH-001: Module-level Rich Console in library code

Projects that start as scripts and are later refactored into importable libraries often retain module-level `Console()` instances that emit output when library functions are called. This breaks the library contract (callers cannot suppress output) and interferes with test capture. Proposed check:

```yaml
- id: ARCH-001
  domain: architecture
  severity: suggested
  description: "Library module instantiates Rich Console() or print() at module level, coupling import to side effects"
  verify: "grep_absent: src/**/*.py, '^console = Console()'"
  override_eligible: true
  notes: >
    Acceptable pattern: pass a Console instance as a parameter or use a module-level
    logger instead. Override eligible for intentional CLI-only entry-point modules
    (e.g., cli.py) where the module is never imported by other library code.
```

## Agent Scope Expansion Candidates

- **python-toolchain-auditor**: Currently checks for `[tool.interrogate]` presence only. Consider adding a check for `fail-under` threshold value; projects that add the section but set `fail-under=0` (or omit it, accepting the default of 0) satisfy the presence check while gaining no enforcement. A threshold of 70 or higher should be required.

- **pre-commit-auditor**: Currently checks for the presence of a no-em-dash hook but does not verify the regex pattern used. A hook that matches the em-dash character (U+2014) but not `--` (double hyphen used as a substitute) would pass the presence check while missing a common variant. Consider verifying the pattern string, not just hook presence.

- **devops-deployment-agent**: The `|| true` pattern (CI-007a above) is not currently in scope. The agent checks for scanner presence but not for exit-code suppression. Expanding to grep for `|| true` adjacent to known scanner invocations would catch this class of false-assurance findings.

- **ossf-compliance-auditor**: The solo-maintainer branch-protection pattern warrants a note in the check. When `required_approving_review_count` cannot be set to 1 because there is only one maintainer, the correct remedy is to enable branch protection with a bypass rule for the maintainer, not to skip protection entirely. The auditor's remediation guidance should distinguish between "absent entirely" and "present but zero reviews required."

1. **Branch protection on main** (OSSF-004): Enable via `gh api` or the GitHub UI. The remediation command was provided during the session. For a solo-maintainer repo, enable protection with a maintainer bypass rule so force-push to main is still possible when needed. Required before `OSSF-009` (PR review enforcement) can be addressed.

2. **Private Vulnerability Reporting** (OSSF-002 partial): The GitHub API rejected the enable request because the repository is private. This feature is available on public repos and GitHub Enterprise. The SECURITY.md file was created with a contact email fallback. No further action needed unless the repo is made public or migrated to GitHub Enterprise.

3. **Best Practices Badge** (OSSF-001-BADGE, SCORECARD:CII-Best-Practices): Requires manual registration at `bestpractices.coreinfrastructure.org`. No API pathway exists for automated registration. Once registered, the badge URL should be added to README.md.

4. **Signed releases and SBOM** (SCORECARD:Signed-Releases, SCORECARD:SBOM): Both require a release workflow that does not yet exist. A release workflow using `cosign` for signing and `syft` or `cyclonedx-python` for SBOM generation should be created before the first versioned release. The Scorecard workflow will pick up the change automatically once the release workflow is present.

## Follow-on Technical Work

The following items were identified during remediation but deferred. They represent real gaps, not cosmetic issues.

1. **Ruff violation cleanup** (20 new violations from TOOL-006): Adding 13 Ruff rule codes surfaced 20 new violations: `TC001`/`TC002`/`TC003` (TYPE_CHECKING import guards), `S608` (SQL in string literals, likely false positives in `queries.py` that use parameterised queries), and `ASYNC240` in `download.py`. Each category requires a targeted fix: import guards for TC*, inline `# noqa: S608` with a tracking comment for confirmed false-positive SQL strings, and an httpx-compatible async pattern for ASYNC240.

2. **Test coverage gaps**: `download.py` has no unit tests. The CLI commands `load`, `refresh`, and `download` are also untested. Coverage gates currently pass because the covered modules pull the aggregate above 80%, but these modules represent the data pipeline entry points and carry the highest operational risk.

3. **`fetch_isins_batch` docstring**: The docstring for this function describes incorrect parameter semantics. This was flagged by the darglint hook after it was added. Should be fixed as part of the Ruff cleanup pass.

4. **`load` CLI command importing private functions**: The `load` command imports functions that are named with a leading underscore, indicating they are not part of the public API of their module. This should be resolved either by making the functions public or by restructuring the module boundary.

5. **`gleif name` empty string guard**: Passing an empty string to the `name` command produces a query that matches all records. A guard clause or a minimum-length validator on the Typer argument should be added.
